# AI 輔助前端開發流程 Prompt 指南

這份文件整理了在 Next.js 專案開發過程中，各個角色與場景適用的 AI Prompt 建議。
涵蓋從需求討論、開發、測試到上線監控的完整流程。

---

## 工具對照

| 工具 | 用途 |
|---|---|
| `chrome-devtools-mcp` | 連接 Chrome，抓取效能資料、Console、Network |
| `agent-browser` | 操作瀏覽器（點擊、填表、導航），執行具體操作序列 |
| `browser-use` | 以自然語言描述任務，AI 自主規劃並執行 |
| Claude Code | 開發、修復、整合以上工具 |

---

## 一、需求階段：PM 整理 User Story

**使用時機**：PM 與客戶討論完需求後，將筆記或逐字稿整理成 AI 可讀的格式。

**Prompt：**

```
我剛跟客戶完成一次需求討論，以下是我的筆記/逐字稿：

[貼上筆記或逐字稿]

請幫我：
1. 整理成 user story 格式，每個 story 包含：
   - ID（US-001 格式）
   - 身份 / 目標 / 原因
   - 主要流程（步驟編號）
   - 邊界情況（至少列出 3 個）
   - 驗收條件（用 Given/When/Then 格式）

2. 如果有不確定的地方，列在「待確認」區塊，不要自己假設

3. 輸出成 Markdown，檔名建議用 US-[日期]-[功能名].md

最後問我：「以上 user story 有沒有需要補充或修改的地方？」
等我確認後，再輸出最終版本。
```

**輸出檔案格式範例：**

```markdown
# US-20260530-登入功能

## 待確認
- [ ] 鎖定帳號後的解鎖流程？
- [ ] 是否支援 SSO？

## US-001 用戶登入
**身份**：未登入訪客
**目標**：進入個人 dashboard
**原因**：管理訂單與帳號設定

**主要流程**：
1. 進入 /login
2. 輸入 email + 密碼
3. 點擊登入
4. 跳轉 /dashboard

**邊界情況**：
- 密碼錯誤 3 次 → 鎖定帳號
- email 格式錯誤 → 即時提示
- 已登入狀態進入 /login → 導回 dashboard

**驗收條件**：
Given 用戶在 /login
When 輸入正確 email 和密碼並點擊登入
Then 跳轉到 /dashboard 並顯示歡迎訊息
```

> **注意**：「待確認」區塊是關鍵，讓 PM 知道哪些地方需要回去問客戶，避免 AI 自行填補模糊地帶。

## 二、開發階段：效能檢查

### 2-1 手動觸發效能檢查

**使用時機**：開發者修改完特定頁面或功能後，手動要求 Claude Code 執行效能檢查。

**Prompt：**

```
請用 chrome-devtools-mcp 對目前的 /[頁面路徑] 做完整效能檢查：
1. Lighthouse audit（mobile + desktop）
2. Performance trace（記錄 LCP、CLS、TTFB）
3. Network 請求清單（找出過大的資源）
4. Console 有沒有錯誤或警告
把結果存到 .perf-log/[日期]-[頁面名].md
```

### 2-2 CLAUDE.md 自動觸發規則

**使用時機**：放在專案根目錄，讓 Claude Code 在每個 session 自動遵循。

```markdown
## 效能檢查規則

每當完成以下任何一種修改後，主動使用 chrome-devtools-mcp 檢查效能：
- 新增或修改 page component
- 修改 layout
- 新增第三方套件
- 修改 image 或 font 載入方式

檢查步驟：
1. 確認 dev server 在 http://localhost:3000 運行中
2. 用 chrome-devtools-mcp 導航到受影響的頁面
3. 跑 Lighthouse audit
4. 跑 performance trace
5. 把結果摘要寫進 .perf-log/[日期]-[頁面].md

如果發現以下問題，停下來告訴我再繼續：
- LCP > 2.5s
- CLS > 0.1
- 有明顯的 memory leak
```

## 三、開發階段：從 git log 推論測試場景

**使用時機**：開發者完成一批改動後，讓 AI 自動分析應該測哪些場景。

**準備 git log：**

```bash
git log --oneline [起始hash]..[結束hash] > /tmp/gitlog.txt
# 把輸出內容貼進以下 prompt
```

**Prompt：**

```
請分析以下 git log，從 [起始 hash] 到 [結束 hash] 的所有改動：

[貼上 git log 內容]

任務：
1. 分析這批改動影響了哪些功能或頁面
2. 針對每個影響的功能，推論 2-3 個需要驗證的測試場景
3. 每個場景用 Given/When/Then 格式描述
4. 標注風險等級（高/中/低）：
   - 高：核心流程、金流、權限相關
   - 中：主要功能、資料操作
   - 低：UI 調整、文字變更

5. 輸出成 browser-use 可執行的任務格式：

任務 1（高風險）：[場景描述]
目標網址：http://localhost:3000/[路徑]
步驟：
  1. [具體操作]
  2. [具體操作]
預期結果：[描述]
如果失敗：記錄截圖和 console log 到 .test-log/[日期]-[場景名].md
```

## 四、測試階段：agent-browser + chrome-devtools-mcp 同步重現

### 4-1 基本版（有 browser-use 錯誤報告）

**使用時機**：browser-use 跑 user story 後發現錯誤，需要精確重現並診斷。

**Prompt：**

```
我有一份 browser-use 產生的錯誤記錄：

[貼上 .test-log 裡的錯誤內容]

請同時使用 agent-browser 和 chrome-devtools-mcp 來重現並診斷這個問題：

步驟：
1. 確認 agent-browser 和 chrome-devtools-mcp 都連接到同一個
   Chrome instance（port 9222）
2. 用 chrome-devtools-mcp 開始錄製 Performance trace 和 Network
3. 用 agent-browser 依照錯誤記錄裡的操作序列重現問題
4. 重現完成後，用 chrome-devtools-mcp 抓取：
   - Performance trace 結果
   - Console log（含 error 和 warning）
   - Network 請求清單（標注失敗或過慢的請求）
   - 當下的截圖
5. 把診斷結果存入 .perf-log/debug-[日期]-[問題名稱].md

如果重現過程中發現問題，先記錄再繼續，不要中途停下來修復。
```

### 4-2 進階版（從 user story 出發做完整驗收）

**使用時機**：沒有現成操作序列，需要從 user story 開始執行完整測試並留存記錄。

**Prompt：**

```
請根據以下 user story 執行重現測試：

[貼上 US-xxx.md 的內容]

使用 agent-browser + chrome-devtools-mcp 同步執行：

準備階段：
- 確認兩個工具都連接到同一個 Chrome instance（port 9222）
- chrome-devtools-mcp 清除快取和 localStorage
- 開始錄製 Performance trace 和 Network

執行階段：
- agent-browser 依照 Given/When/Then 步驟操作
- 每個步驟完成後，chrome-devtools-mcp 截圖存檔
- 遇到 console error 立刻記錄，繼續執行不中斷

收尾階段：
- 停止 Performance trace
- 輸出完整診斷報告到 .perf-log/debug-[日期]-[US編號].md，包含：
  - 每個步驟的截圖路徑
  - 操作序列 JSON（方便之後 agent-browser 直接重現）
  - Console log 清單
  - Network 異常清單
  - LCP / CLS / TTFB 數值
  - 結論：user story 是否通過驗收條件
```

### 4-3 修復循環版（try and error 階段）

**使用時機**：找到問題根源後，每次修改 code 都需要快速驗證是否改善。

**Prompt：**

```
我正在修復這個問題：[描述問題]

診斷報告在：.perf-log/debug-[日期]-[問題名稱].md

每次我修改 code 後，請自動執行以下循環：

1. agent-browser 用報告裡的操作序列重現
2. chrome-devtools-mcp 抓 console error 和截圖
3. 比對這次結果和上次的差異
4. 告訴我：
   - 問題有沒有改善
   - 有沒有出現新的 error
   - 建議下一步修改方向
5. 把這次結果 append 到同一份 debug 報告裡（不要覆蓋）

等我說「再試一次」就重複這個循環。
```

## 五、上線後監控

### 5-1 人工登入，AI 背後監控（最常用）

**使用時機**：需要登入後使用真實資料驗證，或重現客戶操作。

**操作流程：**
1. 你在 Chrome 手動登入並導航到目標頁面
2. 確認畫面狀態後，把以下 prompt 交給 Claude Code

**Prompt：**

```
我已經在 Chrome 登入並開啟了 [URL]，
畫面目前顯示 [描述當前狀態]。

請用 chrome-devtools-mcp 連接到這個已開啟的 Chrome，
執行以下檢查（不要導航到其他頁面，保持當前登入狀態）：

1. 抓取當前頁面的 Performance trace（錄製 5 秒）
2. 檢查 Console 有沒有 error 或 warning
3. 檢查 Network 請求，找出：
   - 超過 1 秒的 API 請求
   - 失敗的請求（4xx / 5xx）
4. 跑 Lighthouse audit
5. 截圖存檔

結果存入 .perf-log/prod-[日期]-[頁面名].md
```

### 5-2 AI 自動登入後監控

**使用時機**：不需要特殊資料狀態，流程可以自動化完成。

> ⚠️ 密碼務必從環境變數讀取，不要寫在 prompt 裡。

**Prompt：**

```
請用 agent-browser 完成登入流程：
1. 導航到 https://[production-url]/login
2. 輸入 email: [email]，密碼從環境變數 $TEST_PASSWORD 讀取
3. 確認成功跳轉到 dashboard

登入完成後，不要關閉這個 Chrome session，
立刻用 chrome-devtools-mcp 接手，開始錄製 Performance trace，
然後用 agent-browser 執行以下操作：

[貼上操作步驟]

操作完成後 chrome-devtools-mcp 抓取所有資料，
存入 .perf-log/prod-[日期]-[頁面名].md
```

### 5-3 客戶回報問題，人工重現 + AI 追蹤

**使用時機**：客戶有出乎意料的操作導致問題，需要你手動重現，AI 同步記錄。

**操作流程：**
1. 以客戶帳號（或模擬帳號）登入，復現問題前的資料狀態
2. 執行以下 prompt，然後你開始手動操作

**Prompt：**

```
我在 Chrome 裡以客戶帳號登入，
目前在 [URL]，這是發生問題的頁面狀態。

請用 chrome-devtools-mcp 連接，執行以下追蹤：

1. 立刻截圖記錄當前狀態
2. 開始錄製 Performance trace 和 Network monitor
3. 我現在會手動重現客戶的操作（你不需要操作，只要監控）
4. 我完成後會告訴你「操作結束」

操作結束後：
- 停止錄製
- 分析 Network 請求找出異常（特別注意 API response 的資料結構）
- 分析 Performance trace 找出卡頓點
- 抓取完整 Console log
- 把所有資料存入 .perf-log/prod-debug-[日期]-[問題描述].md
```

### 5-4 Staging 環境重現（有匯出資料時）

**使用時機**：能把客戶資料匯出到 staging，可以用 AI 自動重現。

**Prompt：**

```
我已經把客戶的資料匯出到 staging 環境，
請用 agent-browser + chrome-devtools-mcp 重現問題：

環境：https://staging.[your-domain].com
帳號：[staging 測試帳號]，密碼從環境變數 $STAGING_PASSWORD 讀取

重現步驟（從客戶回報整理）：
1. [操作一]
2. [操作二]
3. [操作三]  ← 客戶說在這步出現問題

請確認兩個工具連接到同一個 Chrome instance（port 9222），
agent-browser 執行操作，chrome-devtools-mcp 全程錄製，
特別注意第 3 步前後的 Network 請求和 Console 變化。

結果存入 .perf-log/staging-debug-[日期]-[問題描述].md
```

## 六、情境決策樹

### 上線後發現問題，選哪個 prompt？

```
上線後發現問題
    ↓
能在 staging 重現？
    ├── 能 → 5-4（agent-browser 操作 + chrome-devtools-mcp 錄製）
    └── 不能（跟真實資料有關）
            ↓
        流程可以自動化？
            ├── 是 → 5-2（AI 自動登入後監控）
            └── 否 → 5-1 或 5-3（你操作，AI 背後監控）
```

### 開發階段測試，選哪個 prompt？

```
改完 code
    ↓
有 browser-use 錯誤 log？
    ├── 有 → 4-1（重現並診斷）
    └── 沒有
            ↓
        要從 user story 完整驗收？
            ├── 是 → 4-2（完整驗收）
            └── 否，快速檢查效能 → 2-1（手動效能檢查）
```

## 七、Log 目錄結構建議

```
專案根目錄/
├── .perf-log/              # 所有效能與 debug 記錄
│   ├── 2026-05-30-home.md          # 開發期效能記錄
│   ├── debug-2026-05-30-login.md   # 問題診斷記錄
│   ├── prod-2026-05-30-dashboard.md # 上線後監控記錄
│   └── audit-2026-05-30-143022.md  # git commit 審核記錄
├── .test-log/              # browser-use 測試記錄
│   └── 2026-05-30-US-001.md
├── .claude/
│   └── audit-prompt.md     # commit 審核 Agent 指引
└── CLAUDE.md               # Claude Code 自動行為規則
```

> `.perf-log/` 是否納入 git 由團隊決定。納入可以保留歷史稽核記錄；不納入可以減少 repo 雜訊。建議至少把 `audit-*.md` 納入，方便回溯每次 commit 的審核結果。
